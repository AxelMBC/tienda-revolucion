## 1. Objetivo

Implementar el backend del sistema definido en la unidad 2, capaz de:

- Persistir el catálogo de productos en una base de datos relacional.
- Exponer una superficie mínima de API que el frontend pueda consumir
  (lectura pública) y que el dueño de la tienda pueda administrar
  (escritura autenticada).
- Quedar listo para conectarse al frontend ya existente en la siguiente
  actividad (3.2), reemplazando el catálogo *hard-coded* que vive en
  `lib/products.ts`.

## 2. Alcance acordado

En la actividad de diseño de la unidad 2 se decidió **mantener el alcance
estrecho** para poder terminar dentro del plazo del curso:

| Decisión                | Valor                                                        |
|-------------------------|--------------------------------------------------------------|
| Tipo de backend         | API REST sobre Next.js Route Handlers (sin servicio aparte). |
| Base de datos           | PostgreSQL 16.                                               |
| ORM                     | Prisma 7.8.                                                  |
| Validación de entrada   | Zod en cada endpoint de escritura.                           |
| Hosting de BD producción| Supabase (Postgres administrado).                            |
| Hosting de BD desarrollo| Docker (`postgres:16-alpine`) en `localhost`.                |
| Autenticación admin     | Contraseña única + *cookie* firmada (`jose`).                |
| Hashing                 | `bcryptjs`, factor 12.                                       |
| **Fuera de alcance**    | Carrito persistente, pagos, órdenes, reservaciones, búsqueda full-text, multi-admin. |

La razón es que **la venta termina en WhatsApp** (`lib/whatsapp.ts`):
el cliente arma el pedido, el sitio abre `wa.me` con el mensaje
prellenado y el dueño confirma la venta por chat. Eso elimina la
necesidad de cobros en línea y de un módulo de órdenes.

```

│ Navegador / Cliente    │         │ Dueño (admin)          │
└──────────┬─────────────┘         └──────────┬─────────────┘
           │ GET /api/products                │ POST /api/admin/login
           │ GET /api/producto/[slug]         │ POST /api/admin/products
           ▼                                  ▼
┌──────────────────────────────────────────────────────────┐

│ ─ Zod valida entrada ─ middleware.ts protege /admin/*    │

┌──────────────────────────────────────────────────────────┐

│   prod → Supabase (pooler 6543, directo 5432)            │
└──────────────────────────────────────────────────────────┘
```

## 4. Proceso de desarrollo

El trabajo se llevó en la rama `feat/backend-prep` y se ordenó en dos
lotes (*Batch A* y *Batch B*) tomados de la guía interna
`docs/backend-setup.md`.

### 4.1 Lote A — base de datos, ORM y lectura pública (completado)

1. **Levantar Postgres local en Docker.** Se escribió
   `docker-compose.yml` con la imagen `postgres:16-alpine`, volumen
   nombrado y credenciales de desarrollo.
2. **Crear el proyecto en Supabase** (para producción). Se generó la
   contraseña y se guardaron las dos cadenas de conexión: *pooler*
   (puerto 6543, para serverless) y *directa* (puerto 5432, para
   migraciones).
3. **Instalar Prisma, Zod y `tsx`** con `yarn add`.
4. **Configurar variables de entorno** en `.env.local` (`DATABASE_URL`,
   `DIRECT_URL`, `ADMIN_PASSWORD_HASH`, `ADMIN_SESSION_SECRET`).
5. **Definir el esquema** en `prisma/schema.prisma`: dos modelos
   (`Product` y `ProductVariant`) más dos *enums* (`Category`, `Size`).
   El precio se modeló como entero en pesos para evitar errores de
   redondeo.
6. **Generar y aplicar la primera migración**
   (`20260525035611_init`) contra la base local.
7. **Crear el cliente Prisma como *singleton*** en `lib/prisma.ts` para
   evitar que Next.js abra una conexión nueva en cada *hot reload*.
8. **Escribir el *seed*** (`prisma/seed.ts`) que toma el arreglo
   `PRODUCTS` de `lib/products.ts` y lo inserta con `upsert`, junto con
   sus variantes por talla. Resultado: **15 productos** poblados en
   local.
9. **Capa de validación** (`lib/schemas.ts`): esquemas Zod para
   `ProductCreate`, `ProductUpdate` y `ProductListQuery`, incluyendo
   sub-esquemas tipados para los campos `Json` (`specs`, `shipping`,
   `story`, `quote`).
10. **Rutas públicas de lectura**:
    - `GET /api/products` — listado paginado con filtro por categoría.
    - `GET /api/products/[slug]` — producto individual con sus
      variantes.

### 4.2 Lote B — autenticación y escritura admin (pendiente)

Diseñado pero todavía sin implementar al cierre de esta entrega:

- `lib/auth.ts` con `verifyPassword`, `signSession`, `verifySession`.
- `middleware.ts` en la raíz protegiendo `/admin/*` y `/api/admin/*`.
- Rutas `POST /api/admin/login`, `POST /api/admin/logout`,
  `POST /api/admin/products`, `PATCH /api/admin/products/[slug]`,
  `DELETE /api/admin/products/[slug]`.
- Llamadas a `revalidateTag("products")` después de cada escritura.

> El Lote B se completará en paralelo con la actividad 3.2 (Frontend),
> ya que la pantalla de administración consume las mismas rutas.

## 7. Dificultades encontradas y cómo se resolvieron

El proceso no fue lineal. Tres problemas importantes obligaron a
desviarse de la planeación original.

### 7.1 Conflicto de puerto con Postgres de Windows

**Síntoma.** Al hacer `docker compose up -d` el contenedor levantaba
sin errores, pero las conexiones a `localhost:5432` caían en una
instalación local de Postgres 17, no en el contenedor.

**Causa.** El servicio de Windows `postgresql-x64-17` ya tenía el
puerto 5432 ocupado, y Docker Desktop no avisa cuando un *bind* falla
silenciosamente en Windows.

**Solución.** Se remapeó el puerto del contenedor de `5432:5432` a
`5433:5432` en `docker-compose.yml`, y se actualizó `DATABASE_URL` a
`localhost:5433`. Se documentó el cambio en `docs/backend-setup.md`
para que el siguiente desarrollador lo encuentre.

### 7.2 Prisma 7 introdujo cambios *breaking* respecto a la guía v6

La guía original se escribió esperando Prisma 6, pero al instalar se
recibió la versión **7.8** (recién publicada). Esto rompió cinco
suposiciones de la guía:

| # | Cambio en Prisma 7                                                    | Cómo se resolvió |
|---|------------------------------------------------------------------------|-------------------|
| 1 | El generador ahora se llama `prisma-client` y emite a una ruta custom. | Se configuró `output = "../lib/generated/prisma"` y se cambiaron los `import` para apuntar a `@/lib/generated/prisma/client`. |
| 2 | El campo `url` está **prohibido** en `schema.prisma`. Debe vivir en `prisma.config.ts`. | Se movió la URL al nuevo `prisma.config.ts`. |
| 3 | `prisma.config.ts` no lee `.env.local` automáticamente.                | Se importó `dotenv/config` y se invocó `loadEnv({ path: ".env.local" })` al inicio del archivo. |
| 4 | `prisma migrate dev` ya no ejecuta `generate` implícitamente.          | Se agregó `prisma generate` como paso explícito. |
| 5 | `new PrismaClient()` exige un *driver adapter*; el constructor sin argumentos ya no tipa. | Se instaló `@prisma/adapter-pg` y `pg`, y `lib/prisma.ts` construye el cliente con `new PrismaPg(DATABASE_URL)`. |

Estos cambios consumieron una tarde completa de depuración. Se decidió
documentarlos en una sección **“Prisma v7 deviations”** dentro de
`docs/backend-setup.md` para que sirvan de referencia futura.

### 7.3 Prisma rechaza interfaces tipadas en columnas `Json`

**Síntoma.** El *seed* fallaba con `TS2322` al pasar los campos
`specs`, `shipping`, `story` y `quote` a `prisma.product.create`.

**Causa.** El tipo `Prisma.InputJsonValue` requiere una *index
signature*, cosa que nuestras interfaces tipadas (`ProductSpecs`, etc.)
no tienen, ya que se prefirió mantenerlas estrictas para el frontend.

**Solución.** Se creó un *helper* `asJson<T>()` que hace una conversión
controlada por `unknown` hacia `InputJsonValue`. El tipo estricto sigue
vigente en el resto del código; el ensanchamiento ocurre solo en la
frontera con Prisma. El mismo patrón se aplicará en las rutas de
escritura del Lote B.

### 7.4 Otros aprendizajes menores

- El esquema original colapsaba “tallas existentes” y “stock por talla”
  en un solo arreglo (`Product.sizes: Size[]`). Se separó en la
  relación `ProductVariant` para que cada talla tenga su propio
  inventario sin migraciones futuras.
- Se decidió usar `Int` para el precio (pesos enteros) en vez de
  `Decimal`. El catálogo no usa centavos y la aritmética entera evita
  una clase entera de bugs de redondeo.

## 8. Cambios respecto a la planeación de la unidad 2

| Área           | Planeado                                  | Final                                            | Por qué |
|----------------|-------------------------------------------|--------------------------------------------------|---------|
| ORM            | Prisma 6                                  | Prisma 7.8                                       | Versión disponible al instalar; se documentaron las diferencias. |
| Puerto BD local| 5432                                      | 5433                                             | Conflicto con servicio Windows de Postgres 17.    |
| Modelo `Product` | `sizes: Size[]` + `stock: Int` plano    | Relación `ProductVariant` (talla + stock + default) | Permite stock por talla sin migración futura.     |
| `seed` config  | `package.json → "prisma": { "seed": ... }`| `prisma.config.ts → migrations.seed`             | Prisma 7 ignora el campo en `package.json`.       |

Ninguno de estos cambios alteró el alcance funcional acordado en la
unidad 2 (CRUD de catálogo + auth de un solo dueño); todos fueron
adaptaciones al entorno técnico real.

## 10. Conclusión

El backend cumple con el alcance acordado en la unidad 2 en su parte
de **lectura pública** (Lote A): base de datos modelada, migrada y
sembrada; cliente Prisma robusto contra el *hot reload* de Next; rutas
`GET` paginadas y validadas; y una guía interna
(`docs/backend-setup.md`) que documenta cada desviación.

El **Lote B** (autenticación y escritura admin) está completamente
diseñado y a un par de horas de implementación; se completará durante
la actividad 3.2 porque comparte interfaz con la pantalla de
administración del frontend.

Las dificultades técnicas más costosas (puerto 5432 ocupado y los
cambios *breaking* de Prisma 7) se resolvieron sin reducir el alcance:
solo se ajustaron detalles de implementación. La planeación original
en cuanto a **modelos, endpoints y flujo de venta vía WhatsApp**
permaneció intacta.
